线程

进程和线程的概念

进程:进程是操作系统分配资源的基本单位，也就是操作系统是按进程分配资源的。

线程:线程操作系统调度的基本单位，一个进程至少有一个线程，进程里的线程共享进程的资源。

举一个比喻，进程就好比一间办公室，线程就好比办公室里的员工，办公室拥有分配的资源(空间)，而员工可以共享这些资源，同时利用这些资源来完成日常的工作。

因此操作系统给进程分配了它所需要的资源，由进程中的线程共同利用这些资源来完成任务。

谈一点自己对多线程的理解，即为什么要出现多线程。

首先按照一进程一线程的概念，是完全可以完成任务的，但是假如两个进程需要通信时，需要额外开辟资源给它们用来通信，而且还需要频繁的切换进程，及其影响效率。

以上述的例子举例，假如办公室将每个员工都隔开，每个员工仅仅拥有自己的一点小办公室，那么当员工间需要交流时，就需要起身再找一个空地来进行交流，这显然是很浪费时间的。因此直接将员工放在一个大的办公室内，有问题就可以直接在这个办公室内交流，节省了寻找空房间的时间。

所以，多线程的实质，就是将多个有关联任务放在一个进程里，减少进程切换带来的时间损耗。

线程安全

由于线程共享某些资源和线程执行的随机性，多线程的执行结果是不确定的，这样的多线程是不安全的。

就像办公室的电脑，谁都可以用，一个员工可能在工作，中途出去了，另一个员工没有保存工作结果直接开始工作，中途也出去，回来的员工就直接在上面继续自己的工作，最终的结果是，两个员工的工作都不是自己想要的结果，因为中途都被另一个人修改了。

因此需要一种机制，证明自己正在使用这个资源，以确保别人知道不会来抢占。

线程状态

并发访问

Java在语言层面上提供了synchronized关键字，synchronized关键字就是告诉其他线程的机制，这个资源已经占用了，让其等待。

synchronized同步方法

线程访问synchronized修饰的方法时，会尝试去拿方法的使用权，如果发现有线程正在使用这个方法，就不会进入方法体，而是等待其他线程执行完。

这个使用权在Java里就是加锁，即第一个进入synchronized同步语句的线程获得锁。

synchronized同步语句块

有时候一个方法内还有很多其他操作，而对共享变量操作的代码只是一小部分，可以尝试synchronized修饰关键代码块，这样其他线程访问方法时，可以将方法体中的其他操作先完成，而对共享变量的访问会被拒绝，有效地提高了线程的效率。这里使用synchronized(this)监视的是当前对象，即其他线程进入synchronized修饰的语句时都会被拒绝。

synchronized其他对象

当一个线程进入synchronized语句块后，就持有这个对象的锁，那么其他线程就只能访问这个对象的非synchronized修饰的方法，无法进入任何synchronized修饰的方法。

因此，当对象有大量的synchronized的方法，但是它们操作的不是同一个共享变量时，使用synchronized方法会导致其他无关的synchronized方法也无法获得锁，因此可以使用synchronized对象来区分不同的锁。

synchronized Class

synchronized不仅可以对对象上锁，还可以对类的对象Class上锁，对Class上锁会影响对象所有实例，即使线程访问对象不同的实例的synchronized方法，也会被拒绝。

但是Class锁和对象锁是两种锁，并不影响，即线程获得Class锁，其他线程仍然可以获得对象锁。

1.synchronized非静态方法锁住的是对象实例，但是其他线程依旧可以异步访问非synchronized方法。

2.synchronized静态方法和synchronized(Object.class)一样，是类锁，锁Class文件，对所有类的实例都有效。

3.出现异常，锁自动释放。

4.同步不具有继承性。

5.synchronized锁是可以重入的，即获得对象锁后还能继续获得对象锁。表现为synchronized语句内调用synchronized语句，子类同步方法可以通过可重入锁调用父类同步方法。

总结

对象锁

synchronized可以看作申请锁的过程，当线程进入synchronized语句块时，首先尝试去获得对应的锁，如果synchronized语句块没有被上锁，那么就获得锁，并对其加上锁，发现已经被锁住，那么就等待释放锁。

Class锁

因为对象在堆内，可以有多个不同的实例，因此类的不同对象实例争取自己实例的锁，而类的Class描述类的类，在内存中只有一份，因此类的不同对象实例争取同一个锁。

单例模式与多线程

  1. 立即加载

    类加载就创建实例，能在多线程下保持单例状态，但是并不是在使用的时候创建，会造成内存资源的浪费。

  2. 延迟加载

    延迟加载模式在多线程环境下无法保持单例的状态。

  • 2.1 synchronizedd同步方法

    同步方法获取实例时需要频繁加锁，效率太低。

  • 2.2 synchronizedd同步方法块

    可能有多个线程同时进入else语句块，依旧不能保持单例的状态。

  • 2.3 DCL双检查锁机制

    即使多个线程同时进入else语句块，但是在后续的同步语句中无法通过if语句的检查。
